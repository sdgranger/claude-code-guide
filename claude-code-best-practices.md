# Claude Code 모범 사례 가이드

> 출처: [Claude Code 모범 사례](https://code.claude.com/docs/ko/best-practices)
> 환경 구성부터 병렬 세션 확장까지 Claude Code를 최대한 활용하기 위한 실전 패턴입니다.

---

## 핵심 원칙

> **Context window가 가장 중요한 리소스입니다.**
> 모든 메시지, 읽은 파일, 명령 출력이 컨텍스트를 채웁니다.
> 채워질수록 성능이 저하됩니다 — 이전 지시사항을 "잊기" 시작합니다.

이 가이드의 모든 모범 사례는 이 한 가지 제약 조건에서 출발합니다.

---

## 1. Claude에게 작업을 검증할 방법 제공하기 (가장 높은 영향력!)

Claude가 스스로 작업을 확인할 수 있을 때 훨씬 더 잘 수행합니다.
검증 수단이 없으면 올바르게 **보이지만** 작동하지 않는 코드를 생성할 수 있습니다.

### 검증 전략 비교

| 전략 | 나쁜 예 | 좋은 예 |
|------|---------|---------|
| **검증 기준 제공** | "이메일 검증 함수 만들어" | "validateEmail 함수 작성. user@example.com→true, invalid→false, user@.com→false. 구현 후 테스트 실행해" |
| **UI 시각적 검증** | "대시보드 개선해" | "[스크린샷 붙여넣기] 이 디자인 구현하고, 결과 스크린샷 찍어서 원본과 비교해. 차이점 나열하고 수정해" |
| **근본 원인 해결** | "빌드가 실패해" | "빌드가 이 오류로 실패: [오류]. 수정하고 빌드 성공 확인. 오류를 억제하지 말고 근본 원인 해결해" |

### 검증 수단의 종류

```
┌─────────────────────────────────────────┐
│           검증 방법 선택 가이드            │
├─────────────────────────────────────────┤
│                                         │
│  코드 로직  → 테스트 케이스              │
│  UI 변경   → 스크린샷 비교              │
│  빌드/설정 → 빌드 성공 확인 명령         │
│  API      → curl/httpie로 응답 확인      │
│  코드 품질 → linter 실행                │
│  성능     → 벤치마크 실행 및 수치 비교    │
│                                         │
└─────────────────────────────────────────┘
```

### 프롬프트 템플릿

```
[작업 설명]

검증 방법:
- 구현 후 `npm test` 실행해서 모든 테스트 통과 확인
- 새 기능에 대한 테스트도 작성해
- 기존 테스트가 깨지지 않는지 확인해
```

> **검증에 투자하세요.** 견고한 검증 = Claude가 자율적으로 올바른 결과를 도출

- [ ] 테스트/스크린샷/예상 출력을 포함한 프롬프트 작성 연습
- [ ] "수정 후 테스트 실행해서 통과하는지 확인해" 패턴 사용해보기
- [ ] UI 작업 시 스크린샷 붙여넣기 + 비교 요청해보기

## 2. 먼저 탐색하고, 그 다음 계획하고, 그 다음 코드 작성하기

바로 코딩으로 뛰어들면 **잘못된 문제를 해결하는 코드**가 생성될 수 있습니다.

### 4단계 워크플로우

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  1단계 탐색 (Plan Mode)                                   │
│  ─────────────────────                                   │
│  "src/auth를 읽고 세션/로그인 처리 방식을 이해해"             │
│  → Claude가 파일 읽기만 (변경 없음)                         │
│                      ↓                                   │
│  2단계 계획 (Plan Mode)                                   │
│  ─────────────────────                                   │
│  "Google OAuth를 추가하려면 어떤 파일을 변경해야 해?          │
│   계획을 작성해"                                           │
│  → Ctrl+G로 계획을 에디터에서 직접 편집 가능                  │
│                      ↓                                   │
│  3단계 구현 (Normal Mode)                                 │
│  ──────────────────────                                  │
│  "계획대로 구현하고, 테스트 작성해서 실행해"                    │
│  → Claude가 코드 작성 + 검증                               │
│                      ↓                                   │
│  4단계 커밋                                               │
│  ──────────                                              │
│  "설명적인 메시지로 커밋하고 PR 열어"                         │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 언제 계획을 건너뛸까?

| 건너뛰기 OK | 계획 필요 |
|------------|---------|
| 오타 수정 | 접근 방식이 불확실할 때 |
| 로그 한 줄 추가 | 여러 파일을 수정할 때 |
| 변수 이름 변경 | 수정 중인 코드에 익숙하지 않을 때 |

> **판단 기준**: diff를 한 문장으로 설명할 수 있으면 → 계획 건너뛰기

- [ ] Plan Mode(`Shift+Tab`)로 코드 분석 후 Normal Mode로 전환하여 구현 해보기
- [ ] `Ctrl+G`로 계획을 에디터에서 편집해보기
- [ ] 계획 건너뛰기가 적절한 상황 판단해보기

## 3. 프롬프트에서 구체적인 컨텍스트 제공하기

지시사항이 정확할수록 필요한 수정이 적습니다.

### 4가지 프롬프트 전략

#### 전략 1: 작업 범위 지정
```
나쁜: "foo.py 테스트 추가"
좋은: "사용자가 로그아웃된 엣지 케이스를 다루는 foo.py 테스트 작성.
      mock 피하기. pytest 사용."
```

#### 전략 2: 소스 지적
```
나쁜: "ExecutionFactory가 왜 이상한 API를 가지고 있어?"
좋은: "ExecutionFactory의 git 히스토리를 살펴보고
      API가 어떻게 발전했는지 요약해"
```

#### 전략 3: 기존 패턴 참조
```
나쁜: "캘린더 위젯 추가"
좋은: "HotDogWidget.php 패턴을 따라 캘린더 위젯 구현해.
      월 선택 + 앞뒤 이동으로 연도 선택 가능하게.
      기존 라이브러리만 사용, 외부 라이브러리 없이."
```

#### 전략 4: 증상 설명
```
나쁜: "로그인 버그 수정"
좋은: "세션 타임아웃 후 로그인 실패한다고 보고됨.
      src/auth/ 토큰 리프레시 확인.
      문제를 재현하는 실패 테스트 작성 후 수정해."
```

### 풍부한 콘텐츠 제공 방법

| 방법 | 설명 | 예시 |
|------|------|------|
| `@파일명` | 파일 직접 참조 | `@src/auth/login.ts의 로직 설명해` |
| 이미지 붙여넣기 | `Ctrl+V`로 스크린샷 | UI 디자인 비교, 에러 화면 공유 |
| URL 제공 | 문서/API 참조 | "이 API 문서 참고해서 구현해" |
| 파이프 전달 | 데이터 직접 전송 | `cat error.log \| claude -p "분석해"` |
| Claude에게 가져오게 | Bash/MCP/파일 읽기 | "먼저 관련 코드를 찾아서 분석해" |

> **모호한 프롬프트도 유용할 때가 있습니다:**
> "이 파일에서 무엇을 개선하겠어?" → 생각하지 못한 것을 발견할 수 있음

- [ ] 4가지 전략 중 하나를 적용하여 구체적인 프롬프트 작성해보기
- [ ] `@파일명`으로 파일을 참조하여 질문해보기
- [ ] 에러 로그를 파이프로 전달하여 분석 요청해보기

## 4. 환경 구성하기

### 4.1 효과적인 CLAUDE.md 작성

```bash
/init    # 프로젝트 분석 후 자동 생성
```

#### CLAUDE.md 체크리스트

| 포함할 것 | 제외할 것 |
|-----------|-----------|
| Claude가 추측 불가한 Bash 명령 | Claude가 코드를 읽어 파악 가능한 것 |
| 기본값과 다른 코드 스타일 규칙 | 표준 언어 규칙 (Claude가 이미 앎) |
| 테스트 지시사항 및 선호 테스트 러너 | 상세한 API 문서 (대신 링크) |
| 저장소 에티켓 (브랜치 네이밍, PR 규칙) | 자주 변경되는 정보 |
| 아키텍처 결정 | 긴 설명/튜토리얼 |
| 일반적인 함정 또는 명백하지 않은 동작 | 파일별 코드베이스 설명 |

> **핵심 질문**: *"이 줄을 제거하면 Claude가 실수를 할까?"* 아니라면 삭제!
> 부풀려진 CLAUDE.md = Claude가 중요한 규칙을 무시

#### CLAUDE.md 예시

```markdown
# 코드 스타일
- ES 모듈(import/export) 구문 사용, CommonJS(require) 사용 금지
- 가능하면 import를 구조 분해 (예: import { foo } from 'bar')

# 워크플로우
- 코드 변경 완료 시 타입 체크 확인
- 전체 테스트보다 단일 테스트 실행 선호 (성능)
```

#### import & 중첩

```markdown
# CLAUDE.md에서 외부 파일 참조
@README.md
@docs/git-instructions.md
@~/.claude/my-project-instructions.md
```

> CLAUDE.md를 **git에 체크인**하여 팀이 기여할 수 있도록 하세요.
> 시간이 지남에 따라 가치가 증가합니다.

### 4.2 권한 구성

```bash
/permissions    # 현재 설정 확인
/sandbox        # OS 수준 격리
```

안전한 명령을 허용 목록에 추가하여 승인 요청을 줄이세요:

```json
{
  "permissions": {
    "allow": ["Bash(npm run lint)", "Bash(git commit *)"]
  }
}
```

> 10번째 승인 후에는 검토 없이 클릭만 하고 있습니다 → 허용 목록으로 해결

### 4.3 CLI 도구 활용

```
Claude에게 CLI 도구 사용법을 가르치세요:
"foo-cli-tool --help 로 foo 도구를 배우고, A, B, C를 해결해"
```

| CLI 도구 | 용도 |
|----------|------|
| `gh` | GitHub 이슈, PR, 댓글 |
| `aws` | AWS 서비스 관리 |
| `gcloud` | Google Cloud |
| `sentry-cli` | 에러 모니터링 |

> CLI 도구 = 외부 서비스와 상호작용하는 **가장 컨텍스트 효율적인** 방법

### 4.4 MCP 서버 연결

```bash
claude mcp add --transport http github https://api.githubcopilot.com/mcp/
```

### 4.5 Hooks 설정

```
"모든 파일 편집 후 eslint를 실행하는 hook 작성해"
"마이그레이션 폴더에 대한 쓰기를 차단하는 hook 작성해"
```

> CLAUDE.md 지침 = 권고 (Claude가 판단)
> Hooks = 결정론적 (예외 없이 매번 실행)

### 4.6 Skills 생성

```markdown
# .claude/skills/deploy/SKILL.md
---
name: deploy
description: 배포 프로세스 실행
disable-model-invocation: true
---
$ARGUMENTS 환경에 배포합니다.
1. 테스트 실행
2. 빌드
3. 배포 스크립트 실행
```

### 4.7 Subagents 생성

```markdown
# .claude/agents/security-reviewer.md
---
name: security-reviewer
description: 보안 취약점 코드 검토
tools: Read, Grep, Glob, Bash
model: opus
---
선임 보안 엔지니어로서 검토:
- 주입 취약점 (SQL, XSS, 명령 주입)
- 인증/권한 부여 결함
- 코드 내 비밀/자격 증명
```

- [ ] CLAUDE.md를 위 기준으로 점검하기
- [ ] `/permissions`로 허용 목록 설정하기
- [ ] 프로젝트에 필요한 Skill 또는 Hook 하나 만들어보기

## 5. 효과적으로 소통하기

### 5.1 코드베이스 질문하기

선임 엔지니어에게 물어볼 질문을 Claude에게 하세요:

```
"로깅은 어떻게 작동해?"
"새로운 API 엔드포인트를 어떻게 만들어?"
"foo.rs 134번 줄의 async move { ... }는 무엇을 해?"
"CustomerOnboardingFlowImpl은 어떤 엣지 케이스를 처리해?"
"이 코드가 333번 줄에서 bar() 대신 foo()를 호출하는 이유는?"
```

> 특별한 프롬프팅 불필요. 직접 질문하세요.
> 새 코드베이스 온보딩에 매우 효과적입니다.

### 5.2 Claude에게 인터뷰 받기 (큰 기능)

```
[간단한 설명]을 빌드하고 싶어.
AskUserQuestion 도구를 사용하여 인터뷰해줘.

기술 구현, UI/UX, 엣지 케이스, 트레이드오프에 대해 질문해.
명백한 질문 말고, 내가 고려하지 않았을 어려운 부분을 파고들어.

모든 것을 다루면 SPEC.md에 완전한 사양을 작성해.
```

> 사양 완료 후 → **새 세션** 시작 → 깨끗한 context + 작성된 사양 참조

- [ ] Claude에게 현재 프로젝트에 대해 질문해보기
- [ ] 새 기능 구현 시 인터뷰 패턴 사용해보기

## 6. 세션 관리하기

### 6.1 즉시 방향 수정하기

| 키/명령 | 동작 | 상세 |
|---------|------|------|
| `Esc` | 현재 작업 중단 | context 보존, 방향 전환 가능 |
| `Esc` × 2 / `/rewind` | 체크포인트 복원 | 대화+코드 모두 이전 상태로 |
| `"Undo that"` | 변경사항 되돌리기 | Claude에게 요청 |
| `/clear` | context 완전 초기화 | 관련 없는 작업 전환 시 |

### 6.2 Context 관리 규칙

```
┌─────────────────────────────────────────────────────┐
│                Context 관리 흐름도                    │
│                                                     │
│  작업 전환?  ───yes──→  /clear  (완전 초기화)         │
│      │no                                            │
│      ↓                                              │
│  context 70%+?  ──yes──→  /compact <지시>            │
│      │no                     (유지할 내용 지정)       │
│      ↓                                              │
│  부분 압축 필요?  ──yes──→  Esc+Esc → Summarize      │
│      │no                                            │
│      ↓                                              │
│  빠른 질문?  ──yes──→  /btw  (context 미사용)         │
│      │no                                            │
│      ↓                                              │
│  많은 파일 탐색?  ──yes──→  Subagent 사용             │
│      │no                     (별도 context)          │
│      ↓                                              │
│  계속 작업                                           │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 6.3 Subagent로 조사 위임하기

```
"subagents를 사용하여 인증 시스템이 토큰 리프레시를 어떻게 처리하는지 조사해"
```

```
메인 context: [프롬프트] → [subagent 요약만] → [계속 작업]
                              ↑
Subagent:    [파일1] → [파일2] → ... → [파일20] → [요약 생성]
             (메인 context에 영향 없음! 깨끗하게 유지!)
```

> 구현 후에도 검증을 위해 subagent 사용 가능:
> `"subagent를 사용하여 이 코드를 엣지 케이스에 대해 검토해"`

### 6.4 체크포인트로 Rewind하기

```
Claude의 모든 작업은 체크포인트를 생성합니다.
→ Esc+Esc 또는 /rewind로 rewind 메뉴 열기
→ 대화만 복원 / 코드만 복원 / 둘 다 / 부분 요약

위험한 시도도 안전: 작동하지 않으면 rewind하고 다른 접근 시도!
```

### 6.5 대화 재개하기

```bash
claude --continue         # 가장 최근 대화 재개
claude --resume           # 최근 세션에서 선택
/rename oauth-migration   # 세션에 설명적 이름 지정
```

### 7.1 비대화형 모드

### 7.2 여러 세션 병렬 실행

#### Writer/Reviewer 패턴

| 세션 A (작성자) | 세션 B (검토자) |
|----------------|----------------|
| `API 엔드포인트에 대한 속도 제한기 구현` | |
| | `속도 제한기 구현을 검토해. 엣지 케이스, 경쟁 조건 찾아` |
| `검토 피드백: [세션 B 결과]. 이슈 해결해` | |

> 새로운 context는 Claude가 방금 작성한 코드에 편향되지 않아 **코드 검토 품질 향상**

#### Test-First 패턴

| 세션 A (테스트 작성) | 세션 B (구현) |
|-------------------|-------------|
| 테스트 먼저 작성 | 테스트를 통과하는 코드 작성 |

### 7.3 Fan-out 패턴 (대규모 마이그레이션)

```bash
# 1단계: 대상 파일 목록 생성
claude -p "마이그레이션이 필요한 Python 파일 나열" > files.txt

# 2단계: 2-3개 파일로 프롬프트 테스트
head -3 files.txt | while read file; do
  claude -p "React에서 Vue로 $file 마이그레이션. OK 또는 FAIL 반환" \
    --allowedTools "Edit,Bash(git commit *)"
done

# 3단계: 프롬프트 개선 후 전체 실행
cat files.txt | while read file; do
  claude -p "$file 마이그레이션 실행" \
    --allowedTools "Edit,Bash(git commit *)"
done
```

> `--allowedTools`로 Claude가 할 수 있는 작업을 제한 (무인 실행 시 중요!)

## 8. 흔한 실패 패턴과 해결법

### 실패 패턴 진단표

| 실패 패턴 | 증상 | 근본 원인 | 해결법 |
|-----------|------|---------|--------|
| **주방 싱크 세션** | 한 세션에서 여러 관련 없는 작업 | 관련 없는 context 축적 | 작업 전환 시 **`/clear`** |
| **반복적 수정** | 같은 문제 2번+ 수정 시도 | 실패한 접근 방식으로 context 오염 | **`/clear`** 후 배운 내용 통합한 더 나은 프롬프트 |
| **과도한 CLAUDE.md** | Claude가 규칙 무시 | 규칙이 노이즈에 묻힘 | 무자비하게 정리. 불필요한 항목 삭제 또는 Hook으로 전환 |
| **신뢰-검증 간극** | 그럴듯하지만 작동하지 않는 코드 | 검증 수단 없음 | 항상 검증 제공 (테스트/스크립트/스크린샷) |
| **무한 탐색** | Claude가 수백 개 파일 읽으며 context 소진 | 범위 미지정 | 조사를 좁게 범위 지정하거나 **Subagent** 사용 |

### 가장 중요한 규칙

> **더 나은 프롬프트가 있는 깨끗한 세션** >>>
> **누적된 수정이 있는 긴 세션**

```
실패 루프에 빠졌을 때:

1. /clear 실행
2. 지금까지 배운 것 정리
3. 더 구체적인 프롬프트 작성
4. 새로 시작

이것이 3번째 수정 시도보다 거의 항상 빠릅니다!
```

## 9. 직관 개발하기

이 가이드의 패턴은 **시작점**이지 절대 규칙이 아닙니다.

### 때로는 규칙을 깨야 할 때

| 규칙 | 깨야 할 때 |
|------|-----------|
| context를 깨끗하게 유지 | 복잡한 문제에서 기록이 가치 있을 때 |
| 계획 먼저 | 작업이 탐색적이어서 Claude가 파악하게 할 때 |
| 구체적인 프롬프트 | Claude가 문제를 해석하는 방식을 보고 싶을 때 |

### 직관을 기르는 방법

```
Claude가 훌륭한 결과를 낼 때:
→ 프롬프트 구조, 제공한 context, 사용한 모드를 기억하세요

Claude가 어려움을 겪을 때:
→ 왜? context가 시끄러웠나? 프롬프트가 모호했나?
  작업이 한 번에 너무 컸나?
```

> 시간이 지남에 따라 **구체적일 때와 개방적일 때**,
> **계획할 때와 탐색할 때**, **context를 지울 때와 누적할 때**를
> 아는 직관을 개발하게 됩니다.

> **전체 요약 — Claude Code 모범 사례 핵심 5가지**
>
> 1. **검증 수단을 제공하라** — 테스트, 스크린샷, 예상 출력
> 2. **탐색 → 계획 → 구현** — 바로 코딩하지 말고 먼저 이해하라
> 3. **구체적으로 프롬프트하라** — 파일, 제약 조건, 패턴 참조
> 4. **Context를 관리하라** — `/clear`, `/compact`, Subagent 활용
> 5. **실패 루프를 인식하라** — 2번 수정 실패 시 `/clear` 후 재시작
